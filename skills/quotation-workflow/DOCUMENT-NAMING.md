# 📄 文档命名规范

**版本：** v2.0  
**更新日期：** 2026-03-30  
**适用范围：** 所有对外发送的商务文档

---

## 🎯 核心原则

1. **简洁专业** - 文件名简短清晰，无多余后缀
2. **统一格式** - 所有文档遵循同一命名规则
3. **易于识别** - 通过文件名即可识别文档类型和内容

---

## 📋 命名规则

### 基本格式

```
<文档类型>-<日期>-<序号>.pdf
```

**示例：**
- `QT-20260330-001.pdf` - 报价单
- `PI-20260330-001.pdf` - 形式发票
- `SC-20260330-001.pdf` - 销售合同

---

## 📁 文档类型代码

| 代码 | 英文全称 | 中文名称 | 用途 |
|------|----------|----------|------|
| **QT** | Quotation | 报价单 | 产品价格报价 |
| **PI** | Proforma Invoice | 形式发票 | 预开发票，用于收款 |
| **SC** | Sales Contract | 销售合同 | 正式销售合同 |
| **CI** | Commercial Invoice | 商业发票 | 正式发票，用于清关 |
| **PL** | Packing List | 装箱单 | 包装明细 |
| **SO** | Sales Order | 销售订单 | 内部订单确认 |
| **PR** | Payment Reminder | 催款单 | 付款提醒 |
| **AR** | Aging Report | 账龄报告 | 应收账款报告 |

---

## 📅 日期格式

**格式：** `YYYYMMDD`（8 位数字）

**示例：**
- `20260330` - 2026 年 3 月 30 日
- `20260401` - 2026 年 4 月 1 日

---

## 🔢 序号规则

**格式：** 3 位数字，从 `001` 开始

**示例：**
- `001` - 当天第 1 份
- `002` - 当天第 2 份
- `015` - 当天第 15 份

**序号重置：** 每天从 `001` 重新开始

## ⚠️ 常见错误

### 错误 1: 多余的后缀

```bash
# ❌ 错误
QT-20260330-001-HTML.pdf
QT-20260330-001-Final.pdf
QT-20260330-001-v2.pdf

# ✅ 正确
QT-20260330-001.pdf
```

**原因：** 内部版本信息不应出现在对外文件名中

---

### 错误 2: 日期格式不统一

```bash
# ❌ 错误
QT-2026-03-30-001.pdf
QT-03302026-001.pdf
QT-2026033001.pdf

# ✅ 正确
QT-20260330-001.pdf
```

**原因：** 日期格式必须为 `YYYYMMDD`（8 位）

---

### 错误 3: 序号位数不统一

```bash
# ❌ 错误
QT-20260330-1.pdf
QT-20260330-01.pdf
QT-20260330-0001.pdf

# ✅ 正确
QT-20260330-001.pdf
```

**原因：** 序号必须为 3 位数字

## 📊 文件版本管理

### 内部版本

```
QT-20260330-001.json          # 数据源文件（可修改）
QT-20260330-001-v2.json       # 内部版本 2
QT-20260330-001-v3.json       # 内部版本 3
```

### 对外版本

```
QT-20260330-001.pdf           # 对外发送（唯一正式版本）
```

**规则：**
- 对外只发送最终版本的 PDF
- 内部版本用 `-v2`, `-v3` 等后缀区分
- 客户永远看不到内部版本号
